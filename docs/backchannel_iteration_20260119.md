# 相槌タイミング改善の記録（2026-01-19）

このメモは、相槌が発話を邪魔する問題に対して、何を考えて、どう実装して、どうなったかを残すためのものです。あとで手戻りして別案を試すときに、同じところで迷わないことを目的にします。

## 1. 背景と手戻りしたい地点

最初の困りごとは、頷きや首振りの合図が来ると、話し手の発話の途中でも相槌がすぐ鳴ってしまい、割り込みに聞こえることでした。見た目としては、WAITが多すぎたり、保留にしかならなかったり、逆に待機がほとんど発生しないなど、挙動が安定しませんでした。

手戻りしたい地点は、次の指示が出たあたりです。

相槌が常に3秒待つと遅く感じるので、文脈を見ながら即時に返すときは即時に返し、文末まで待てそうなときは待つようにしたい。ログを見ながら改善点を考える。

この地点に戻るときは、今回の実装が何を改善し、何を悪化させたかを明確にしておくのが大事です。

## 2. 目標をどう置いたか

議論の中で、待ち時間を伸ばすことは一時的な改善に見えても本質解決になりにくい、という認識が強くなりました。理由は単純で、相槌の自然さは秒数よりも、文の途中か終わりかをちゃんと理解して動けるかに依存するからです。

そこで、目標を次のように整理しました。

- 相槌を出すなら、基本は区切りで出す（割り込みを減らす）
- 区切りを待てない状況だけ即時にする（即時の価値を残す）
- モデルに渡す材料を、文末判断に向く形にする（ここが本質）

## 3. 実装でやったこと（具体）

ここは「やったこと」を事実として書きます。評価は次章で書きます。

### 3.1 早期呼び出しに切り替えた（締め切り直前では呼ばない）

IMUの合図が来たら、少しだけ待ってからモデルを呼ぶ方式にしました。待つ時間は 0.1 秒程度で、ログ上は `decision_point=signal_early` が出ます。

狙いは、合図が来た時点で「今すぐ返すか」「区切りまで待つか」を決めることでした。以前のように、区切りが来ないときに締め切り直前だけ呼ぶ方式だと、体感の遅れが大きく、しかも会話の区切りとは関係なく鳴りやすいからです。

関連: `app/runtime/session.py`

### 3.2 WAIT を作り、区切りで返す予約を入れた

モデルが WAIT を選んだとき、すぐ再生せずに「区切りが来たら再生する」予約を pending に保存するようにしました。区切りイベント（`segment_end`）が来たら予約を armed にして、相槌を鳴らします。

ログには `planned_set` と `planned_armed` と `backchannel_play planned=true` を出すようにして、後から追えるようにしました。

関連: `app/runtime/session.py`

### 3.3 スキーマエラー（wait_ms 必須）を直した

一度、モデル呼び出しで JSON スキーマのエラーが出ました。`wait_ms` が required に入っていない、という内容です。スキーマ側で `required: ["action", "wait_ms", "reason"]` にして、WAIT以外でも `wait_ms=0` を必ず出す方針にしました。

関連: `app/agents/backchannel_graph.py`

### 3.4 trace を JSONL で出すようにした

改善を続けるために、表示ログだけではなく、機械的に追えるログを出すようにしました。`--trace-jsonl` で JSONL を出力し、モデルに渡した timing や transcript_context、IMU要約、選択結果、理由などを保存します。

関連: `app/runtime/trace.py`, `app/runtime/session.py`

### 3.5 tilt は廃止した

tilt は邪魔なので完全に廃止しました。検出のヒント（gesture_hint）に tilt を割り当てず、相槌の対象も nod/shake のみにしました。

関連: `app/imu/signal.py`, `app/runtime/session.py`

### 3.6 文字起こしを「実際のSTTっぽい」単位に寄せようとした

最初の `transcribe.txt` は2分ごとの長い行で、区切りがほとんど来ませんでした。その結果、区切り待ちの挙動を試せず、締め切り側の判断ばかりになります。

そこで、タイムスタンプがない平文も読み込めるようにして、1行ずつを短いチャンクとして扱い、文字数から再生時間を概算して t_sec を振る方式を入れました。さらに、2分版から短い行を作るスクリプトも作り、`transcribe_stt.txt` として残しました。

関連: `app/transcript/timeline.py`, `scripts/make_transcript_stt_like.py`, `transcribe_stt.txt`

## 4. 実験で見た結果（ログベース）

ここは「ログから言えること」に限定します。

### 4.1 区切りでの判断が一度も発生していない

`data/logs/trace_stt_like2.jsonl` では、`agent_call` が4回ありましたが、すべて `is_boundary=false` で、`decision_point=signal_early` でした。つまり、モデルが区切りで呼ばれた回がありません。

この状態だと、文末で返す経路が実質的に試せていません。

### 4.2 WAIT が選べない状態になっている

同じログで `wait_allowed` は 4/4 が false でした。結果としてモデル出力の `decision.action` はすべて RESPOND_NOW になっています。WAITが候補に無いので、文末待ちに寄せたくても寄せられません。

### 4.3 いまのセグメント残り時間が長い

`segment_remaining_s` は最小でも 3.509 秒でした（中央値は約 4.822 秒、最大 9.56 秒）。この残りが長いほど、即時に返すか見送るかの2択になりやすく、文末待ちの効果が出ません。

## 5. 考察（なぜ本質解決になっていないか）

ここからは意見です。ただし、根拠は上のログと実装の形に置きます。

一番の問題は、モデルが文末を理解できていないというより、モデルが文末を判断できる材料を渡せていないことだと思います。いまは再生中の文を全文で渡しがちで、話していない後半まで見えてしまいます。これだと、モデルは途中か終わりかを判定しにくいです。人間でも、未来の文を先に見たら、今どこを話しているかは分かりません。

もう一つは、設計として WAIT が成立しない条件が多すぎることです。wait_allowed が false だと WAIT は選べません。つまり、モデルが良い判断をしても、実装がその判断を許しません。ここは、モデルの理解以前に、制御の選択肢を閉じてしまっています。

最後に、現状の文字起こしは「実際のSTTの流れ」に近づけたい意図がありましたが、区切り（チャンク）と発話の区切り（意味の切れ目）が一致していない可能性が高いです。このズレがあると、区切りで返すほど不自然になることがあります。区切りを増やすか減らすかより、区切りが何を意味しているかを揃える必要がありそうです。

## 6. 次に試す価値が高い案（手戻り後の方向）

時間が限られる前提で、次の順が良いと思います。

1つ目は、モデル入力を「いま聞こえたところまで」にすることです。再生の経過時間を使って、セグメント全文ではなく途中までの文字列を作って渡します。これができると、モデルは文末かどうかを判断しやすくなります。

2つ目は、再生中の RESPOND_NOW を強く抑えることです。区切りが遠いときは即時に返すのではなく、SKIP を選ばせるほうが自然に感じる可能性があります。即時に返す価値があるのは、区切りが近いときに限る、というルールでもよいです。

3つ目は、WAIT を「常に出せる選択肢」に戻し、上限だけで縛ることです。wait_allowed のような硬いゲートを減らし、モデルの判断を実装が受け止められる形にします。ログで planned_set と planned_armed が増えるかを見て、初めて文末待ちが評価できます。
